# Инструкция: 3 слоя анализа текста через Base

> Полное руководство по использованию Layer A (Rules) + Layer B (Skill) + Layer C (MCP) для анализа **любого** текста — методологического, философского, документации, плана, отчёта.

**Применимо ко всем РП:** РП 6 (статья), РП 3 (ФКЧ-launch), Ш5 (пакет), будущие РП, документация, планы.

---

## Быстрый старт (2 минуты)

### Сценарий 1: Вы пишете текст и хотите автоматическую проверку

```
Вы: "Проверь этот абзац"

Я (Claude):
  1. Автоматически применяю Layer A (Rules) — анализирую через FPF/SPF линзу
  2. Ищу нарушения (A.7, A.1.1, B.5, etc)
  3. Выдаю результат: "Абзац нарушает A.7 — смешали Role и Method"
```

**Результат:** ✅ Быстрая обратная связь (30 сек)

---

### Сценарий 2: Нужна структурированная полная проверка

```
Вы: "/base-lint мой первый абзац"

Я (Claude):
  1. Запускаю Layer B (Skill base-text-lint)
  2. Проверяю по чеклисту:
     ✅ Distinctions: A.7
     ⚠️ Bounded Context: A.1.1
     ✅ Method form: A.3.1
     ❌ Work Products: A.15.1
  3. Выдаю таблицу нарушений + рекомендации
```

**Результат:** 📊 Структурированный отчёт (5-10 мин)

## Layer B: Skill base-text-lint — полная инструкция

### Как это работает

**Статус:** ✅ Готово, используется по команде `/base-lint`

### Пошаговая инструкция

#### Шаг 1: Подготовьте текст

**Ваше действие:** Откройте файл (черновик) и решите, что проверить:
- Весь документ
- Один раздел (например, "§1 Сущность и Понятие")
- Один абзац

#### Шаг 2: Вызовите команду /base-lint

**Ваше действие:** Напишите мне команду:

```
/base-lint второй абзац про Aufhebung
```

или

```
/base-lint раздел 'Логика снятия'
```

или просто копируете текст:

```
/base-lint [вставляете текст]
```

#### Шаг 3: Что происходит внутри (мой процесс)

**Я запускаю Skill base-text-lint:**

1. Загружаю FPF справку из `memory/fpf-reference.md`
2. Анализирую ваш текст по FPF паттернам:
   - **Distinctions** (A.7): Role ≠ Method ≠ Work — разделены ясно?
   - **Bounded context** (A.1.1): Термины определены локально?
   - **Method form** (A.3.1): Это действие (как?) или описание (что?)?
   - **Work products** (A.15.1): Ясен артефакт?
   - **ADI cycle** (B.5): Логика: Abduction → Deduction → Induction?
3. Собираю нарушения в таблицу с FPF кодами
4. Форматирую результат в стандартный отчёт

#### Шаг 4: Получите структурированный отчёт

**Артефакт (то, что вы получите):**

```
# Base-Text-Lint Report

Text analyzed: "Абзац про Aufhebung" (lines 45-52)
Lint level: Standard

## Summary
- 🔴 CRITICAL: 1 (distinctions)
- 🟡 WARNING: 2 (bounded context, method form)
- ✅ OK: 2 (work products, reasoning)

## Findings

### 🔴 CRITICAL

Finding 1: Role and Method mixed
- FPF: A.7
- Text: "Aufhebung — это методологический процесс отрицания"
- Issue: Keyword 'методологический' (method) applied to 'процесс' (process/description)
- Fix: "Aufhebung — метод: (1) выявить противоречие, (2) разрешить синтезом"

### 🟡 WARNING

Finding 2: Bounded context unclear
- FPF: A.1.1
- Text: "В логике снятия мы используем..."
- Issue: Term 'логика снятия' not defined locally
- Fix: Add definition: "В этом документе логика снятия означает: ..."

Finding 3: Method ambiguous (action or description?)
- FPF: A.3.1
- Text: "Снятие происходит через отрицание"
- Issue: Passive voice — not clear if action or natural process
- Fix: "Метод: активно применить отрицание к противоречию"

## Lint Stats
| Category | Violations | FPF codes |
|----------|-----------|-----------|
| Distinctions | 1 | A.7 |
| Bounded Context | 1 | A.1.1 |
| Method form | 1 | A.3.1 |
| Work Products | 0 | — |
| Reasoning | 0 | B.5 |
| TOTAL | 3 | — |
```

**Что это означает:**
- 🔴 CRITICAL = срочно исправить (блокирует качество)
- 🟡 WARNING = желательно улучшить (лучше, но не критично)
- ✅ OK = хорошо, нарушений нет

#### Шаг 5: Переделайте текст

**Ваше действие:** Используйте рекомендации и переделайте текст.

**Результат:** Улучшенный текст.

#### Шаг 6: Повторите проверку

**Ваше действие:** Вызовите `/base-lint` ещё раз:

```
/base-lint [переделанный текст]
```

**Результат:** Новый отчёт. Если все зелёное ✅ → текст готов.

## Интеграция с любым РП (универсальный процесс)

### Общая схема для ВСЕХ рабочих продуктов

**ЭТАП 1: НАПИСАНИЕ**
- Вы пишете черновик (любого РП)

**ЭТАП 2: БЫСТРАЯ ПРОВЕРКА (Layer A)**
```
Вы: "Проверь этот текст"
Я применяю Layer A (Rules):
  ✓ Distinctions чётко разделены?
  ✓ Bounded context определён?
  ✓ Методы как действия?
→ Быстрая обратная связь (30 сек)
```

**ЭТАП 3: ГЛУБОКАЯ ПРОВЕРКА (Layer B)**
```
Вы: "/base-lint [текст]"
Я запускаю Skill:
  ✓ Полный чеклист FPF паттернов
  ✓ Таблица нарушений с кодами
  ✓ Конкретные рекомендации
→ Структурированный отчёт (5-10 мин)
```

**ЭТАП 4: ПЕРЕДЕЛКА**
- Переделываете по рекомендациям
- Вернётесь к ЭТАПУ 2 (Layer A)
- Повторяйте, пока не будет ✅

**ЭТАП 5: ФИНАЛЬНАЯ ПРОВЕРКА**
```
Вы: "/base-lint вся работа"
Результат: Если все зелёное ✅ → можно коммитить/отправлять
```

## Когда использовать /base-lint

✅ Закончили писать абзац → `/base-lint` перед продолжением
✅ Переписали по рекомендациям → `/base-lint` снова (верификация)

✅ При сомнениях в структуре → `/base-lint` проблемное место
✅ Хотите таблицу нарушений → всегда `/base-lint`

## FAQ

### Q: Какая разница между Layer A и Layer B?

A:
- **Layer A:** Быстро, неформально. Я отвечаю импровизированно, используя FPF фреймворк.
  ```
  Вы: "Проверь абзац"
  Я: "Нарушает A.7 — смешали Role и Method"
  ```

- **Layer B:** Структурированно, полно. Я запускаю чеклист и выдаю таблицу.
  ```
  Вы: "/base-lint абзац"
  Я: Таблица с кодами FPF и рекомендациями
  ```

**Используйте Layer A** во время работы.
**Используйте Layer B** перед финалом.

### Q: Что если я не согласен с рекомендацией?

A: Обсудим! Я ссылаюсь на FPF код → вместе проверяем, может быть, ваш контекст требует исключения.

### Q: Должен ли я вызывать /base-lint каждый раз?

A: Нет, используйте:
- **Layer A часто** (во время работы)
- **Layer B иногда** (перед финалом)
