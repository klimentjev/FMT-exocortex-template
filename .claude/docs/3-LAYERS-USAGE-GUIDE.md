# Инструкция: 3 слоя анализа текста через Base

> Полное руководство по использованию Layer A (Rules) + Layer B (Skill) + Layer C (MCP) для анализа философских и методологических текстов.

---

## Быстрый старт (2 минуты)

### Сценарий 1: Вы пишете текст и хотите автоматическую проверку

```
Вы: "Проверь этот абзац статьи через Base"

Я (Claude):
  1. Автоматически применяю Layer A (Rules) — анализирую через FPF/SPF линзу
  2. Ищу нарушения (A.7, A.1.1, B.5, etc) в вашем тексте
  3. Выдаю результат: "Абзац нарушает A.7 — смешали Role и Method"
```

**Результат:** ✅ Быстрая обратная связь (30 сек)

---

### Сценарий 2: Нужна структурированная полная проверка

```
Вы: "/base-lint мой первый абзац о Aufhebung"

Я (Claude):
  1. Запускаю Layer B (Skill base-text-lint)
  2. Проверяю по чеклисту:
     ✅ Distinctions: A.7
     ⚠️ Bounded Context: A.1.1
     ✅ Method form: A.3.1
     ❌ Work Products: A.15.1
  3. Выдаю таблицу нарушений + рекомендации
```

**Результат:** 📊 Структурированный отчёт (5 мин)

### Сценарий 3: Нужна информация из Base (будущее — Layer C MCP)

```
Я (Claude): "Мне нужно определение A.7, дайте из Base"

MCP (Layer C):
  1. Автоматически ищет в Base/FPF-Spec.md § A.7
  2. Возвращает: определение + примеры + anti-patterns
  3. Я использую в анализе текста
```

**Статус:** ⏳ Готовится к использованию (W15)

## Layer A: Cursor Rules (автоматический анализ)

### Как это работает

**Статус:** ✅ Активна всегда (встроена в мои инструкции)

**Срабатывает:** Когда вы показываете мне философский или методический текст

### Примеры использования

#### Пример 1: Простой вопрос

```
Вы: "Проверь эту фразу: 'Aufhebung — это методология отрицания'"

Я (применяю Layer A Rules):
  ↓
  Вижу: FPF A.7 триггер → Role ≠ Method
  ↓
  Ответ: "Фраза нарушает A.7. 'Методология' — это Method (как делать), 
  но вы описываете Object (что такое Aufhebung). 
  Переделайте: 'Aufhebung — метод: (1) найти противоречие, (2) разрешить синтезом'"
```

#### Пример 2: Проверка bounded context

```
Вы: "В статье я использую слово 'логика'. Где я его определил?"

Я (применяю Layer A Rules, A.1.1):
  ↓
  Вижу: термин не определён локально
  ↓
  Ответ: "Нарушена A.1.1 (Bounded Context). 'Логика' используется без определения. 
  Добавьте: 'В этом документе логика означает: система правил для...' (ваше определение)"
```

#### Пример 3: Проверка на hermeneutic vs normative

```
Вы: "Статья описывает, как Гегель использовал снятие"

Я (применяю Layer A Rules, FPF E.4):
  ↓
  Вижу: hermeneutic (описание существующего) vs normative (норма для присвоения)
  ↓
  Ответ: "Это hermeneutic текст (что было у Гегеля). Но ваша РП 6 требует 
  normative форму (как присвоить метод). Переделайте на: 'Метод Aufhebung — это...'"
```

### Когда Layer A срабатывает автоматически

✅ Вы показываете текст статьи
✅ Вы спрашиваете о различении (Role, Method, etc)
✅ Вы говорите "проверь" или "анализируй"
✅ Я сам предлагаю: "Это может нарушать A.7"

### Быстрая справка для Layer A

Используйте эту таблицу, если хотите напомнить мне о конкретном паттерне:

| Если вы говорите | Я проверяю (Layer A) | FPF код |
|-----------------|----------------------|---------|
| "Проверь это различение" | Разделены ли Role, Method, Work? | A.7 |
| "Это правильно определено?" | Определено ли локально в bounded context? | A.1.1 |
| "Как это должно быть методом?" | Это действие (как?) или описание (что?) | A.3.1 |
| "Что это должно создавать?" | Какой артефакт (работа) это производит? | A.15.1 |
| "Логика аргумента правильная?" | Abduction → Deduction → Induction? | B.5 |
| "Это может меняться?" | Есть ли триггер эволюции? | B.4 |

## Layer B: Skill base-text-lint (структурированная проверка)

### Как это работает

**Статус:** ✅ Готово, используется по команде

**Срабатывает:** Когда вы или я вызовём `/base-lint`

### Как использовать: пошаговая инструкция

#### Шаг 1: Подготовьте текст

Откройте файл статьи (черновик) и решите, хотите ли вы проверить:
- Весь документ
- Один раздел (например, "§1 Сущность и Понятие")
- Один абзац

#### Шаг 2: Вызовите команду

```
Вы: "/base-lint второй абзац про Aufhebung"
```

или

```
Вы: "/base-lint раздел 'Логика снятия' (строки 45-78)"
```

#### Шаг 3: Я запускаю Skill

```
Claude запускает Layer B (base-text-lint):

Step 1. Загружаю FPF справку из memory/fpf-reference.md
Step 2. Прохожу по каждому предложению:
        - Distinctions OK?
        - Bounded context OK?
        - Method form OK?
        - Work products OK?
        - ADI cycle OK?
Step 3. Собираю нарушения в таблицу
Step 4. Вывожу результат структурированно
```

#### Шаг 4: Получите отчёт

```
# Base-Text-Lint Report

Text analyzed: "Абзац про Aufhebung" (lines 45-52)
Lint level: Standard

## Summary
- 🔴 CRITICAL: 1 (distinctions)
- 🟡 WARNING: 2 (bounded context, method form)
- ✅ OK: 2 (work products, reasoning)

## Findings

### 🔴 CRITICAL

Finding 1: Role and Method mixed
- FPF: A.7
- Text: "Aufhebung — это методологический процесс отрицания"
- Issue: 'методологический' (method) applied to 'процесс' (process description)
- Fix: "Aufhebung — метод: (1) выявить противоречие, (2) разрешить синтезом"

### 🟡 WARNING

Finding 2: Bounded context unclear
- FPF: A.1.1
- Text: "В логике снятия мы используем..."
- Issue: 'логика снятия' не определена локально
- Fix: Add definition: "В этом документе логика снятия означает: ..."

Finding 3: Method ambiguous (action or description?)
- FPF: A.3.1
- Text: "Снятие происходит через отрицание"
- Issue: 'происходит' (passive) не ясно, это действие или природный процесс
- Fix: "Метод: активно применить отрицание к противоречию"

## Lint Stats
| Category | Violations | FPF codes |
|----------|-----------|-----------|
| Distinctions | 1 | A.7 |
| Bounded Context | 1 | A.1.1 |
| Method form | 1 | A.3.1 |
| Work Products | 0 | — |
| Reasoning | 0 | B.5 |
| TOTAL | 3 | — |
```

#### Шаг 5: Используйте рекомендации

Переделайте текст по рекомендациям и повторите:

```
Вы: "/base-lint переработанный второй абзац"
```

Повторяйте, пока не получите ✅

### Примеры команд

```
# Проверить одну фразу
/base-lint "Aufhebung — это методология"

# Проверить абзац (копируете текст)
/base-lint
[вставляете абзац]
[пустая строка]

# Проверить раздел файла
/base-lint раздел 'Сущность и Понятие' (основной текст РП 6)

# Проверить весь черновик
/base-lint вся статья philosophy-as-logic-of-aufhebung.md

# Проверить с опциями (если нужно)
/base-lint --section 02-entities текст про роли
```

### Когда использовать Layer B

✅ Закончили писать абзац → `/base-lint` перед тем, как продолжать
✅ Переписали по рекомендациям → `/base-lint` снова (верификация)
✅ Перед финальным коммитом статьи → `/base-lint` весь текст
✅ При сомнениях в структуре → `/base-lint` проблемное место
✅ Хотите таблицу нарушений → всегда `/base-lint`

## Layer C: MCP base-search-mcp (автоматический поиск)

### Статус

⏳ **Сейчас:** Stub (инструкции готовы, backend в разработке на W15)
🚀 **Когда готово:** W15 (после реализации Python backend)

### Как будет работать (когда готово)

#### Сценарий 1: Я нужну информацию

```
Я: "Нужно мне определение FPF A.7 для анализа текста"

MCP (Layer C):
  ↓ вызывает get_fpf_pattern("A.7")
  ↓ ищет в Base/FPF/FPF-Spec.md
  ↓ возвращает полный раздел с примерами

Я получаю данные и использую в анализе → более точные рекомендации
```

#### Сценарий 2: Нужен шаблон SPF

```
Вы: "/base-lint, но сравни с SPF template для methods"

Я (через MCP):
  ↓ вызываю search_spf("method-card-template")
  ↓ MCP возвращает Base/SPF/pack-template/03-methods/_method-card-template.md
  ↓ я сравниваю ваш текст с шаблоном
  ↓ вывожу: "Вашему методу не хватает: [что]"
```

#### Сценарий 3: Быстрый поиск по теме

```
Вы: "Найди в Base информацию про 'bounded context'"

MCP (Layer C):
  ↓ search_fpf("bounded context")
  ↓ возвращает все места в Base/FPF/ где это упоминается
  ↓ я даю вам выборку + ссылки

Вы: Получаете готовый материал для быстрого чтения
```

### Как подготовиться к Layer C (сейчас)

1. **Знайте, что MCP будет делать:**
   - `search_fpf(query)` — ищет текст в FPF-Spec.md
   - `get_fpf_pattern(code)` — получает полное определение (A.7, B.5, etc)
   - `search_spf(query)` — ищет в SPF документации
   - `get_spf_template(section)` — получает шаблон (pack, methods, entities, etc)
   - `get_sync_status()` — проверяет версию FPF

2. **Когда реализуется (W15):**
   - Я буду использовать MCP автоматически, не спрашивая вас
   - Вы можете явно: "используй MCP для поиска X в Base"

## Часто задаваемые вопросы

### Q: Как Layer A отличается от Layer B?

A:
- **Layer A (Rules):** Я вам отвечаю импровизированно, используя фреймворк FPF. Быстро, неформально.
  ```
  Вы: "Проверь абзац"
  Я: "Нарушает A.7 — смешали Role и Method"
  ```

- **Layer B (Skill):** Я запускаю структурированный чеклист и выдаю таблицу.
  ```
  Вы: "/base-lint абзац"
  Я: Таблица из 5 пунктов с кодами FPF и рекомендациями
  ```

**Используйте Layer A** для быстрой обратной связи (во время написания).
**Используйте Layer B** для финальной проверки (перед commit).

### Q: А если я забуду, какой слой использовать?

A: Не важно! Просто скажите "проверь мой текст" → я выберу:
- Если небольшой текст → Layer A
- Если большой или просишь /base-lint → Layer B
- Если просишь информацию → будущее Layer C

### Q: Слой C готов использовать?

A: Нет, ещё нет. Реализуется на W15 (Python backend для MCP). Сейчас есть структура и документация.

### Q: Что если я не согласен с рекомендацией?

A: Обсуждаем! Я ссылаюсь на FPF код → вместе проверяем, правильно ли я интерпретирую. Может быть, ваш контекст требует исключения.

### Q: Должен ли я вызывать Layer B каждый раз?

A: Нет, используйте:
- **Layer A часто** (во время работы, неформально)
- **Layer B иногда** (перед commit, для структурированной проверки)
- **Layer C когда готов** (когда нужна справка из Base)
