# Day Open — после «Требует внимания», перед шагом 7 (запись)

> Hook: `day-open.after`. Governance-репо по умолчанию: `DS-strategy` (корень рядом с `IWE`).

## БЛОКИРУЮЩЕЕ: санитация `current/` перед новым DayPlan

Выполнить **до** создания файла `current/DayPlan YYYY-MM-DD.md` и **до** коммита. Без этого шага Day Open **не считать завершённым**.

### 1. Дневные планы

1. Перечислить все файлы `DayPlan *.md` в `{GOVERNANCE_REPO}/current/`.
2. Для каждого файла сравнить дату в имени (`YYYY-MM-DD`) с **сегодняшней** датой (авторитетно: поле «Today's date» в контексте сессии).
3. Если дата в имени **строго раньше** сегодня — **перенести** файл в `{GOVERNANCE_REPO}/archive/day-plans/` (не только «вчера»: переносятся все просроченные, чтобы не копились хвосты).

### 2. Недельные планы

1. Перечислить все `WeekPlan *.md` в `{GOVERNANCE_REPO}/current/`.
2. Для каждого файла прочитать из frontmatter (или шапки) поля **`date_start`** и **`date_end`**.
3. Если **сегодня** не входит в интервал `[date_start; date_end]` — файл **перенести** в `{GOVERNANCE_REPO}/archive/week-plans/`.
4. Канон после шага: в `current/` остаётся **ровно один** `WeekPlan`, чей интервал содержит сегодня. Если таких 0 или >1 — **не закрывать** Day Open: явно зафиксировать в «Требует внимания» и привести структуру в порядок (создать/переименовать / перенести вручную по решению пользователя).

### 3. Контроль

- Если после переносов в `current/` остаётся больше одного актуального WeekPlan или любой `DayPlan` с датой ≠ сегодня — **исправить**, затем продолжать шаг 7 скилла Day Open.
- Коммит governance-репо включает **и** переносы из п.1–2, **и** новый DayPlan (одним или несколькими коммитами — без оставления устаревших планов в `current/`).

### Замена плейсхолдера

`{GOVERNANCE_REPO}` = фактический путь Pack/DS governance (у автора: `DS-strategy` от корня рабочей области `IWE`).
