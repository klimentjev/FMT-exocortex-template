# Рабочая лестница уровней описания системы

> Шаблон в согласии с холонным подходом, ZP, FPF и SPF.
> Нужен, когда надо различить: смысл системы, её модель, холонную архитектуру, операции, процессы, роли, должности, конкретные исполнения и наблюдаемые данные.

## Когда использовать

- При проектировании новой системы, контура или подсистемы
- При разборе путаницы "роль vs должность", "архитектура vs процесс", "модель vs кейс"
- При раскладке документов по правильным уровням
- При проверке, не смешаны ли в одном описании разные типы содержания

## Базовый принцип

`смысл -> модель -> холонная архитектура -> работа системы -> процедуры -> роли управления -> должностная реализация -> конкретные экземпляры -> данные наблюдения`

## 9 уровней

| Уровень | Что там описываем | Чего там не должно быть |
|---------|-------------------|--------------------------|
| `0. Онтологический / смысловой` | Зачем система существует; какое у неё целевое событие, ценность, предмет и базовые инварианты | Оргструктуры, должности, регламенты шагов, локальные кейсы |
| `1. Концептуальный / модельный` | Ключевые сущности, различения, типы объектов, словарь, причинную логику, базовые отношения | Конкретных исполнителей, расписаний, инструкций "кто сегодня что делает" |
| `2. Холонно-архитектурный` | Систему как holon: надсистема, границы, контуры, части, bounded contexts, интерфейсы, роль каждого целого в надсистеме | Микрошагов процессов, должностных инструкций, ситуативных исключений |
| `3. Операционный` | Как система реально работает: циклы, входы-выходы, артефакты, сигналы, ритмы, критерии приёмки, точки управления | Штатного расписания, персональных особенностей людей, UI-шагов |
| `4. Процессный / процедурный` | Процессы, маршруты, сценарии, handoff'ы, условия переходов, протоколы действий | Подмены архитектуры процессом; попытки через регламент определить сущность системы |
| `5. Ролевой / governance` | Функциональные роли, зоны ответственности, права решений, владельцев контуров, эскалации; `role != holder` | Автоматической привязки роли к одному человеку; смешения роли с должностью, агентом или инструментом |
| `6. Организационный / кадровый` | Должности, оргструктуру, закрепление ролей за людьми и командами, линии подчинения, штатное оформление | Онтологии системы, архитектурных инвариантов, логики контуров как таковых |
| `7. Инстансный / исполнительский` | Конкретный запуск, кейс, когорту, занятие, решение, смену, проект, окно времени | Выведения общей архитектуры из единичного случая |
| `8. Наблюдаемость / данные` | Метрики, события, журналы, индикаторы, подтверждающие материалы, факты исполнения | Подмены данных интерпретацией; смешения измерения с нормой или теорией |

## Критические различения

- `Роль != Должность`: роль задаёт функцию и ответственность; должность закрепляет это в организации
- `Архитектура != Процесс`: архитектура задаёт устойчивые части и связи; процесс задаёт ход работы
- `Операционный уровень != Инстансный уровень`: цикл системы не равен конкретному запуску
- `Данные != Модель`: наблюдаемость не создаёт онтологию, а лишь даёт материал для вывода
- `Контур != Подразделение`: контур задаётся функцией и границей ответственности, а не штатной единицей

## Быстрая привязка

- `Процесс` в основном живёт на уровнях `3-4`
- `Роль` в основном живёт на уровнях `2-5`
- `Должность` живёт на уровне `6`
- `Кейс / запуск / когорта` живёт на уровне `7`
- `Метрики / сигналы / логи` живут на уровне `8`

## Мини-шаблон раскладки

```md
## [Название системы]

- Уровень 0: [зачем система существует]
- Уровень 1: [какие сущности и различения здесь есть]
- Уровень 2: [какие контуры / части / границы / интерфейсы]
- Уровень 3: [какие операционные циклы и сигналы]
- Уровень 4: [какие процессы и протоколы]
- Уровень 5: [какие роли и права решений]
- Уровень 6: [какие должности и оргзакрепление]
- Уровень 7: [какие конкретные экземпляры работы]
- Уровень 8: [какие данные и индикаторы наблюдаются]
```
